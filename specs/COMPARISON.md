## vs. Superpowers

**What it is:** Superpowers (predecessor to bigpowers) is a 14-skill methodology for TDD-disciplined AI-agent development. It focuses on implementation and review phases, ships with a session-start bootstrap hook that auto-invokes the first skill, and enforces hard behavioral gates ("Do NOT write code until design is approved") alongside red-flag tables that name the dangerous rationalizations agents commonly make. PR acceptance rate: 6% (94% rejection).

**Where bigpowers aligns:**
- Same core philosophy: spec-driven, TDD-first, evidence over claims
- Same naming convention: two-word verb-noun pairs
- Same subagent model: fresh context per task, two-stage review gate
- Same key skills: `develop-tdd` ↔ `test-driven-development`, `dispatch-agents` ↔ `dispatching-parallel-agents`, `audit-code`/`request-review` ↔ `requesting-code-review`/`receiving-code-review`
- Both embed F.I.R.S.T rubric for tests
- Both produce structured output docs in `specs/`
- **Auto-bootstrapping:** Both now support `session-start` hooks to inject project context automatically.
- **Visual Brainstorming:** Both support a browser-based companion for diagrams and mockups.

**Where bigpowers leads:**
- Full lifecycle: bigpowers covers discover → design → plan → initiate → execute → harden → bug → review → integrate → sustain; superpowers covers only impl + review
- More skills in every phase (39 vs. 14)
- `visual-dashboard` is more generic than superpowers' companion: visualizes architecture (`deepen-architecture`) and plans (`plan-work`) as interactive roadmaps.
- `plan-work` with mandatory `verify: cmd` per step — nobody else does this explicitly
- `kickoff-branch` / `guard-git` / `hook-commits` for project setup automation
- `inspect-quality` / `organize-workspace` for sustain phase
- Multi-harness distribution (Gemini + Cursor extensions generated by `sync-skills.sh`)
- `seed-conventions` generates `CLAUDE.md` + `CONVENTIONS.md` + `specs/` from templates

**Where bigpowers lags / gaps:**
- No HARD-GATE markers: superpowers uses bold "Do NOT proceed" blocks with named violations; bigpowers relies on prose instructions
- No red-flag tables: superpowers names the exact rationalizations agents make (e.g., "This is too simple to need design"); bigpowers doesn't
- Superpowers skills are shorter and more scannable (< 150 KB); some bigpowers skills carry heavy reference docs

**Borrowable ideas:**
- Add HARD-GATE callout blocks to skills that have critical gates (e.g., `plan-work`: "HARD GATE — Do NOT write any code before specs/PLAN.md exists")
- Add a "red flags" section to `develop-tdd` listing common agent rationalisations ("tests are already comprehensive", "this refactor is out of scope")

## vs. BMAD-METHOD

**What it is:** BMAD (Break-through Method for Agile AI-Driven Development) is a 50+ skill ecosystem with a full SDLC from business analysis to deployment. Its defining feature is named agent personas — Mary the Analyst, John the PM, Winston the Architect, Amelia the Developer — each with a personality, role, and menu of capabilities. Agents are customizable via hierarchical TOML (base → team → user). A "party mode" allows multiple personas to collaborate in one session.

**Where bigpowers aligns:**
- Both cover the full lifecycle (analysis through release)
- Both use `CONVENTIONS.md` as the code guardrail loaded by all agents
- Both produce structured artifacts in a `specs/`-equivalent directory
- Both support multiple harnesses (Claude, Cursor, Codex)
- Both emphasise TDD in the implementation phase
- Both use `gh pr create` for releases (not manual PR)

**Where bigpowers leads:**
- Verify-first planning: every step in `specs/PLAN.md` has a `verify: cmd`; BMAD has no equivalent
- Spec-driven development as the unifying thread: bigpowers treats `specs/` as the project's long-term memory; BMAD treats planning artifacts as phase-gated hand-offs
- Simpler model: no persona overhead, no TOML configuration hierarchy, no menu system
- `investigate-bug` / `diagnose-root` / `validate-fix` bug trilogy is more complete than BMAD's dev agent
- Git workflow automation (`kickoff-branch`, `guard-git`, `hook-commits`, `commit-message`, `release-branch`)

**Where bigpowers lags / gaps:**
- No business analysis phase: BMAD has Mary (analyst), market research, competitive landscape, product brief; bigpowers starts at `elaborate-spec` (assumes idea is already validated)
- No UX/design phase: BMAD has a UX designer persona, PRD creation, UX validation; bigpowers has no product design skills
- No sprint planning: BMAD tracks sprint status; bigpowers has no iteration management
- No customisation mechanism: BMAD agents are configurable without forking; bigpowers skills require editing `SKILL.md`
- No documentation site: BMAD has docs.bmad-method.org with tutorials and how-to guides; bigpowers has README only

**Borrowable ideas:**
- A lightweight `to-prd` skill that synthesises an existing conversation into a PRD without an interview (Pocock has this; BMAD has it too)
- A `scope-work` extension that validates market/user assumptions before committing to a feature scope
- Consider a `customise-skill` mechanism (a TOML or YAML overlay that overrides specific fields without editing the source `SKILL.md`)

## vs. Clean Code (Robert C. Martin, 2008)

**What it is:** Clean Code is the definitive book on craft-level software quality. Its core arguments: functions should be small and do one thing; names should be meaningful and intention-revealing; comments explain why, not what; error messages include the offending value; tests follow TDD with one concept per test; classes are small and follow SRP; code should read like well-written prose. The book closes with a complete catalogue of smells and heuristics.

**Where bigpowers aligns:**
- `CONVENTIONS.md` encodes the core Clean Code rules: functions 4–20 lines, files under 500 lines, SRP, early returns over nested ifs, exception messages include offending value, DI over globals, no code duplication
- `audit-code` applies SOLID principles + CONVENTIONS.md as the review checklist
- `enforce-first` applies F.I.R.S.T rubric (which Martin coined)
- Two-word verb-noun naming for skills mirrors Martin's naming discipline
- `develop-tdd` implements Martin's TDD loop: RED → GREEN → REFACTOR
- `model-domain` / `define-language` implement Ubiquitous Language (Evans + Martin overlap)
- Comments convention in `CONVENTIONS.md`: WHY not WHAT; docstrings on public functions; no obvious comments

**Where bigpowers leads:**
- Bigpowers is operational, not descriptive: it tells agents what to do step by step, not principles to internalise
- The `verify: cmd` pattern is a bigpowers invention; Clean Code has no equivalent enforcement mechanism
- Multi-phase lifecycle: Clean Code covers only coding craft; bigpowers wraps it in discovery, planning, and release

**Where bigpowers lags / gaps:**
- Clean Code's heuristics catalogue (G1–G35, N1–N7, T1–T9, etc.) is not surfaced as a checklist in `audit-code`; the skill references SOLID and CONVENTIONS.md but not the full heuristics list
- "Boy Scout Rule" (leave code cleaner than you found it) is referenced in `enforce-first` but not elevated as a first-class habit across all skills
- No dedicated "code smell" recognition skill; the smells are implicit in `audit-code`

**Borrowable ideas:**
- Add Clean Code's heuristics catalogue as a reference doc inside `audit-code/` (similar to how `develop-tdd` has `tests.md`, `mocking.md`, etc.)
- Elevate Boy Scout Rule as an explicit check in `audit-code` and `respond-review`

## vs. Akita's Article (Clean Code for AI Agents, 2026)

**What it is:** A blog post adapting Clean Code principles for LLM-based agents. Core argument: "code is now written once, read dozens of times — by agents." The 13 rules are a direct map of Clean Code priorities reordered for AI context windows: small functions (#1), SRP (#2), meaningful unique names (#3), comments with WHY (#4), explicit types (#5), DRY (#6), runnable tests (#7), predictable directory structure (#8), dependency injection (#9), shallow nesting (#10), contextual error handling (#11), consistent formatting (#12), no redundant comments (#13).

**Where bigpowers aligns:**
- All 13 rules are present in bigpowers either in `CONVENTIONS.md` or embedded in skills:
  - #1 small functions → CONVENTIONS.md (4–20 lines)
  - #2 SRP → CONVENTIONS.md + `audit-code`
  - #3 meaningful unique names → CONVENTIONS.md ("Specific, unique names; avoid `data`, `handler`, `Manager`")
  - #4 comments with WHY → CONVENTIONS.md ("Write WHY not WHAT")
  - #5 explicit types → CONVENTIONS.md (language-agnostic; implied by DI convention)
  - #6 DRY → CONVENTIONS.md ("No code duplication; extract shared logic")
  - #7 runnable tests → `enforce-first`, `develop-tdd`, `plan-work` verify commands
  - #8 predictable directory structure → `seed-conventions` generates canonical layout
  - #9 DI → CONVENTIONS.md ("Inject via constructor/parameter, not globals")
  - #10 shallow nesting → CONVENTIONS.md ("max 2 indentation levels, early returns")
  - #11 contextual error handling → CONVENTIONS.md ("Exception messages include offending value + expected shape")
  - #12 consistent formatting → `hook-commits` enforces linter/formatter at commit time
  - #13 no redundant comments → CONVENTIONS.md ("No obvious comments that restate code")

**Where bigpowers leads:**
- Bigpowers enforces these rules operationally through hooks and skills; the article only recommends them
- `seed-conventions` generates `CONVENTIONS.md` per project so the rules travel with the codebase
- `hook-commits` mechanically enforces formatting/linting; the article relies on agent discipline
- Full lifecycle context: the article addresses only implementation-phase code quality

**Where bigpowers lags / gaps:**
- The article explicitly acknowledges context window constraints as the motivation for all 13 rules; bigpowers has `terse-mode` but no explicit guidance on writing agent-friendly code (code that's easy for agents to read and navigate)
- No "agent-readability" lens in `audit-code`: reviews check human code quality, not whether the code is easy for a future agent to understand

**Borrowable ideas:**
- Add an "agent-readability" checklist to `audit-code`: are functions small enough to fit in one context window? Are names unique enough to be `grep`-able? Are types explicit enough to avoid inference?

## vs. Karpathy's CLAUDE.md (multica-ai/andrej-karpathy-skills)

**What it is:** A single skill containing four behavioral principles distilled from Andrej Karpathy's public observations about LLM coding failure modes. Designed to be merged into any project's `CLAUDE.md`. The four principles: (1) **Think Before Coding** — surface assumptions, present multiple interpretations, push back on overcomplication; (2) **Simplicity First** — minimum viable code, no speculative abstractions; (3) **Surgical Changes** — touch only what the task requires, match existing style; (4) **Goal-Driven Execution** — transform imperative tasks into verifiable success criteria, loop until verified.

**Where bigpowers aligns:**
- `elaborate-spec` covers Think Before Coding (dialogue to refine idea before code)
- `grill-me` / `grill-with-docs` cover assumption surfacing
- `CONVENTIONS.md` covers Simplicity (functions 4–20 lines, no duplication, SRP)
- `plan-work` covers Goal-Driven Execution (every step has `verify: cmd`)
- `validate-fix` / `define-success` cover the verify loop
- `audit-code` + `enforce-first` partially cover Surgical Changes (Boy Scout rule, CONVENTIONS.md compliance)

**Where bigpowers leads:**
- More complete lifecycle: Karpathy's skill is a behavioral overlay for any codebase; bigpowers is an opinionated full workflow
- Specific skills for each phase rather than one set of meta-principles
- Operational enforcement (hooks, verify commands) vs. attitudinal guidelines

**Where bigpowers lags / gaps:**
- No explicit "Think Before Coding" gate baked into execution skills: `develop-tdd` jumps to planning; there is no mandatory pause to surface assumptions before the first line of code
- No "Surgical Changes" check: no skill explicitly asks "are you touching only what the task requires, and nothing else?"
- No "multiple interpretations" step: Karpathy's skill requires presenting 2–3 readings of an ambiguous request before proceeding; bigpowers has `elaborate-spec` / `grill-me` but they are separate skills, not embedded gates in execution
- Karpathy's skill is frictionless to adopt (paste into CLAUDE.md); bigpowers requires install.sh and the full harness

**Borrowable ideas:**
- Add a "Pause and Scope" gate at the start of `develop-tdd`: before planning, state what you will and will not change, and confirm scope with the user if ambiguous
- Add a "Surgical Changes" check to `audit-code`: "Does this diff touch only what the task required?"
- Publish a lightweight `bigpowers-core.md` (analogous to Karpathy's skill) for projects that want the principles without the full install

## vs. Matt Pocock's Skills (mattpocock/skills)

**What it is:** 18 skills (engineering + productivity + misc) built on the same foundation as bigpowers — vertical slices, grilling, TDD — but extended with stronger architecture discipline (referencing John Ousterhout's "A Philosophy of Software Design") and tighter issue tracker integration. Key distinctive skills: `improve-codebase-architecture` (deep modules audit), `zoom-out` (explain code in system-wide context), `triage` (GitHub/Linear issue state machine), `to-issues` (break specs into independent issues), `caveman` (75% token reduction mode). Uses `CONTEXT.md` as the domain glossary and ADRs as decision log, both as first-class artifacts.

**Where bigpowers aligns:**
- Nearly identical foundation: `develop-tdd` ↔ `tdd`, `grill-me` ↔ `grill-me`, `grill-with-docs` ↔ `grill-with-docs`, `investigate-bug` ↔ `diagnose`, `dispatch-agents` ↔ `dispatching-parallel-agents`
- Same vertical-slice philosophy in TDD and task breakdown
- Same CONTEXT.md + ADR pattern (bigpowers: `model-domain`, `define-language`; Pocock: `grill-with-docs` writes CONTEXT.md inline)
- Same two-word verb-noun naming convention
- Same subagent model with fresh context per task
- Same emphasis on runnable verification commands

**Where bigpowers leads:**
- Larger coverage: 39 skills vs. 18; bigpowers has full initiate, harden, and sustain phases
- Git automation: `kickoff-branch`, `guard-git`, `hook-commits`, `commit-message`, `release-branch`; Pocock has only `setup-pre-commit`
- Multi-harness distribution: Gemini + Cursor extensions auto-generated; Pocock is Claude-centric
- `scope-work` → `slice-tasks` → `plan-work` planning trilogy more complete than Pocock's `to-prd` + `to-issues`
- Bug trilogy (`investigate-bug` → `develop-tdd` → `validate-fix`) more explicit than Pocock's `diagnose`

**Where bigpowers lags / gaps:**
- No `improve-codebase-architecture`: Pocock's skill surfaces shallow modules, proposes deepening using vocabulary from "A Philosophy of Software Design" (module depth, leverage, locality, seams); bigpowers has no equivalent
- No `zoom-out`: Pocock's skill asks an agent to explain a piece of code in its full system context before touching it; bigpowers relies on the agent's existing context
- No `triage` or `to-issues`: Pocock integrates with GitHub/Linear; bigpowers keeps everything in `specs/*.md` (deliberate choice, but limits project management integration)
- No `caveman` mode: Pocock's skill reduces output by 75% by dropping filler language; bigpowers has `terse-mode` but it's less prescriptive about token savings
- `handoff` (Pocock) compacts the current conversation into a handoff doc for the next agent; bigpowers has no equivalent (closest is `specs/PLAN.md`, but it's task-scoped, not session-scoped)

**Borrowable ideas:**
- `zoom-out` as a new bigpowers skill: "Before touching this code, explain how it fits into the system — its purpose, callers, dependencies, and invariants"
- `improve-codebase-architecture` as a new bigpowers skill in the Sustain phase, referencing Ousterhout's deep-module vocabulary
- Extend `terse-mode` with Pocock's `caveman` mode rules (drop articles, filler words, pleasantries; compress to 75% of normal length)
- `handoff` as a Utility skill: compact the current conversation + open decisions into a self-contained document the next agent can start from cold

## Synthesis: Bigpowers Strengths

These patterns appear across multiple comparisons as genuinely distinctive to bigpowers:

**1. Verify-first planning.**
Every step in `specs/PLAN.md` must have a `verify: <cmd>` — a runnable command that proves the step succeeded. No other reference (superpowers, BMAD, Clean Code, Pocock) has an equivalent enforcement mechanism. This is bigpowers' most defensible design decision.

**2. Proactive "Bootstrap" Context.**
The `session-start` hook automatically runs a project survey, ensuring the agent is "context-ready" before the user types a single character. This matches superpowers' automation while adding Bigpowers-specific lifecycle mapping.

**3. Visual Command Center.**
The `visual-dashboard` skill provides browser-based visualization of architecture, implementation plans, and UI mockups, closing the parity gap with superpowers' visual companion.

**4. Full-lifecycle span with consistent spec thread.**
Only bigpowers and BMAD cover the full arc from idea to sustain. But bigpowers does it without persona overhead or TOML configuration, using a single `specs/` directory as the project's persistent memory that accumulates across every phase.

**5. Naming discipline.**
Two-word verb-noun pairs (Clean Code's naming applied to skill names) are shared with superpowers and Pocock, but bigpowers enforces it most consistently across all 39 skills.

**6. Multi-harness reach.**
Bigpowers generates Gemini extensions and Cursor rules from the same skill source via `sync-skills.sh`. No other reference supports four harnesses from a single source of truth.

**7. Git lifecycle automation.**
`kickoff-branch` + `guard-git` + `hook-commits` + `commit-message` + `release-branch` form a complete, automated git workflow. Superpowers and Pocock have fragments; BMAD has none.

**8. CONVENTIONS.md as project-level guardrail.**
`seed-conventions` generates a project-specific `CONVENTIONS.md` that all agents read before coding. This makes quality rules travel with the project, not just with the skill system.
