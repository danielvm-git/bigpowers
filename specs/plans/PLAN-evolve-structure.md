## 1. Executive Summary

Evolve the 61 bigpowers skills to produce the **Evolved bigpowers** target structure (ranked 50.0/50.0 in SDD adequacy) instead of the current Original bigpowers structure (33.5/50.0). This requires updating ~35 skills across 7 change themes. The upgrade is **breaking** — new projects adopt the evolved structure; existing projects need migration.

The two documents (`projected-structure-bigpowers-evolved.md` and `sdd-adequacy-ranking.md`) serve as the **requirements** for this plan. They are NOT generated by bigpowers — they describe the target end-state a project achieves after bigpowers processes it.

### Theme C: Epic Capsule Directories

**What changes:** Epics move from flat YAML files (`specs/epics/eNN-slug.yaml`) to capsule directories (`specs/epics/eNN-slug/`) containing an `epic.yaml` manifest, story `.md` specs, decoupled `-tasks.yaml` files, and optional epic-local `adr/` subfolder.

**Epic manifest format** (`epic.yaml`):
```yaml
id: e01
title: "Auth System"
wsjf: 4.5
total_bcps: 8
status: in_progress
stories:
  - id: e01s01
    title: "Login"
    bcps: 3
    status: completed
    spec: e01s01-login.md
    tasks: e01s01-tasks.yaml
  - id: e01s02
    title: "JWT"
    bcps: 5
    status: todo
    spec: e01s02-jwt.md
    tasks: e01s02-tasks.yaml
```

**Story spec format** (`.md`): Fully populated [countable-story-format.md](file:///Users/danielvm/Developer/bigpowers/countable-story-format.md) with 20 required sections and Gherkin acceptance criteria in §17.

**Tasks format** (`-tasks.yaml`):
```yaml
story_id: e01s01
title: "Self-serve password reset"
status: todo
bcps: 3
tasks:
  - id: 1
    description: "Add User model validation tests"
    verify: "npm test -- user-validation.test.ts"
    status: completed
  - id: 2
    description: "Implement uniqueness validation"
    verify: "npm test"
    status: todo
```

### Theme E: State Write-Lock (`state.yaml.lock`)

**What changes:** Any skill that writes to `specs/state.yaml` must acquire a file-based write lock (`specs/state.yaml.lock`) before modification to prevent concurrency conflicts during parallel agent execution.

**Lock protocol:**
1. Before writing `state.yaml`, check for `state.yaml.lock`.
2. If lock exists, read PID/agent-id inside; if stale (>60s), acquire.
3. Write agent ID + timestamp to lock file.
4. Modify `state.yaml`.
5. Remove lock file.

| # | Skill | Change |
|---|-------|--------|
| E1 | `session-state` | ACQUIRE lock before writes to `state.yaml`; document protocol in SKILL.md |
| E2 | `build-epic` | ACQUIRE lock before updating `epic_cycle` / `handoff` in `state.yaml` |
| E3 | `execute-plan` | ACQUIRE lock before writing `decisions[]` or `handoff` to `state.yaml` |
| E4 | `develop-tdd` | ACQUIRE lock before any `state.yaml` update |
| E5 | `verify-work` | ACQUIRE lock before marking story done in `execution-status.yaml` |
| E6 | `release-branch` | ACQUIRE lock before writing `metrics` to `state.yaml` |
| E7 | `kickoff-branch` | ACQUIRE lock before writing `git.branch` / `git.hash` |

**Verification:** `ls specs/state.yaml.lock` does NOT exist after all agent processes complete (lock is always released).

## 9. Migration Path for Existing Projects

For projects currently using Original bigpowers (33.5):

1. **Backup** current `specs/` directory
2. Run `bash scripts/convert-legacy.sh` which:
   - Renames `specs/requirements/` → `specs/product/`
   - Renames `specs/plans/` → `specs/tech-architecture/`
   - Converts `specs/epics/*.yaml` to capsule directories with `epic.yaml` + extracted story `.md` files
   - Creates `specs/verifications/`
   - Creates `specs/epics/archive/`
   - Generates `specs/bugs/registry.yaml` from existing `BUG-*.md` files
   - Moves existing ADRs to correct scope (global or epic-local)
3. Update `CLAUDE.md` to reference new paths
4. Run `bash scripts/validate-specs-yaml.sh` to verify migration
