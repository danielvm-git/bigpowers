# BUG-2026-06-24T045323: npm install fails during postinstall — docs/ excluded from tarball

## Problem

`npm install bigpowers` (v2.28.0+) fails during `postinstall` with:

```
grep: /home/.../bigpowers/docs/references/model-profiles.md: No such file or directory
scripts/generate-reference-tables.sh: line 57: /home/.../bigpowers/docs/references/model-profiles.md: No such file or directory
```

**Actual:** Install exits with code 1. Package is uninstallable through standard `npm install` for downstream consumers (including `pi install npm:bigpowers`).

**Expected:** `npm install bigpowers` succeeds. The postinstall scripts that need `docs/` should either handle its absence or skip when the package is consumed (not in a development checkout).

**Reproduce:** `npm install bigpowers@2.28.0` (or current latest) on any machine that doesn't have a git checkout.

## Root Cause Analysis

### Phase 1 — Reproduce

The issue reporter observed:
```
$ tar tzf bigpowers-2.28.0.tgz | grep -c '^package/docs/'
0
```

Confirmed locally: `npm pack --dry-run` lists 341 entries, zero from `docs/` (the false positives are skill directory names containing "gri**ll-with-docs**", not the `docs/` directory).

### Phase 2 — Isolate

The failure chain:

1. `.npmignore` contains `docs/` → the directory is excluded from the published tarball
2. `package.json` `postinstall` → `bash scripts/sync-skills.sh && bash scripts/install.sh`
3. `sync-skills.sh` last step → unconditionally calls `bash scripts/generate-reference-tables.sh`
4. `generate-reference-tables.sh` sets `TARGET="$REPO_ROOT/docs/references/model-profiles.md"`
5. Line 57: `... >> "$TARGET"` — redirect fails because `docs/` directory doesn't exist
6. Script has `set -euo pipefail` → the `>> ` failure aborts the entire postinstall

**Confirmation:** `npm pack --dry-run` outputs zero `docs/` entries.

### Phase 3 — Hypothesize

| # | Hypothesis | Falsification |
|---|------------|---------------|
| H1 | `.npmignore` `docs/` entry is intentional (docs are dev-only) but `generate-reference-tables.sh` doesn't gate for dev-only context | Add a `.git` guard — if `docs/` is excluded from tarball, the script should not run during consumer install |
| H2 | `generate-reference-tables.sh` should self-heal: create directory and file before writing | Create `mkdir -p "$(dirname "$TARGET")"` and initialize file before writing |
| H3 | Both: the script was never designed to handle a missing target file — it assumes prior git checkout has the file | Both guards combined cover all scenarios |

### Phase 4 — Verify

**Confirmed:** The `.npmignore` `docs/` exclusion is intentional (docs are development reference material, not runtime artifacts). The pre-built `.pi/skills/` and `.pi/prompts/` are what consumers need, and syncing docs references is unnecessary outside development.

**Root cause:** `scripts/sync-skills.sh` unconditionally calls `generate-reference-tables.sh`, which assumes `docs/references/model-profiles.md` exists. In the published tarball, `docs/` is excluded via `.npmignore`.

**Risk level:** High — blocks all `npm install` consumers including `pi install npm:bigpowers`.

## TDD Fix Plan

Two changes, each independently sufficient, both applied for defense in depth:

1. **RED**: `npm install bigpowers` on a clean consumer environment must succeed
   **GREEN**: Gate `generate-reference-tables.sh` in `sync-skills.sh` behind a dev-only check: only run when the checkout has a `.git` directory (i.e., development context, not consumer install)
   **verify**: `bash -c 'cd /tmp && npm pack "$REPO_ROOT" --pack-destination /tmp && tar xzf bigpowers-*.tgz && cd package && npm install --ignore-scripts && bash scripts/sync-skills.sh 2>&1' | grep -c "reference-tables"` should be 0 when git is absent

2. **RED**: `generate-reference-tables.sh` should not crash when `docs/references/` directory is missing
   **GREEN**: Add `mkdir -p "$(dirname "$TARGET")"` at the top of the script, and guard the `grep -r` call so it still produces valid output when no matches exist (empty table)
   **verify**: `bash scripts/generate-reference-tables.sh` succeeds when `docs/references/` doesn't exist

## Acceptance Criteria

- [ ] `npm install bigpowers` succeeds without `--ignore-scripts` on a clean install
- [ ] `generate-reference-tables.sh` runs successfully in a dev checkout (with `.git` and `docs/` present) — no regression
- [ ] `generate-reference-tables.sh` runs successfully when `docs/references/` directory is missing — creates it
- [ ] `sync-skills.sh` does not call `generate-reference-tables.sh` in a non-git checkout
- [ ] Existing CI workflows that call `sync-skills.sh` still produce valid reference tables

## Resolution

**Fixed.** Two changes:

1. **`scripts/sync-skills.sh`** — `generate-reference-tables.sh` now gated behind `[[ -d .git ]]` so it only runs in development context, not during consumer `npm install`
2. **`scripts/generate-reference-tables.sh`** — added `mkdir -p "$(dirname "$TARGET")"` so the script self-heals even if `docs/references/` doesn't exist

**Verified:**
- ✅ Dev context (with `.git`, `docs/` present): generates reference tables, exit 0
- ✅ Consumer context (no `.git`, no `docs/`): skips reference tables, exit 0
- ✅ Self-heal (`docs/` missing, called directly): creates directory + file, exit 0
