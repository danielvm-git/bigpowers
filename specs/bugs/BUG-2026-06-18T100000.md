# BUG-2026-06-18T100000: YAML frontmatter quoting breaks skill loading

## Problem

Pi v0.79.6 reports 23 skill conflicts at startup — YAML frontmatter in 24 `.pi/skills/*/SKILL.md` files fails to parse. Additionally, 6 other files parse successfully but embed `model: sonnet` inside the description string, making the model hint invisible to Pi.

- **Actual**: 24 SKILL.md files fail YAML parsing; 6 parse but with lost model hints
- **Expected**: All SKILL.md files parse successfully, and `model:` is a separate YAML key where specified
- **Reproduce**: Start Pi in the bigpowers repo; observe [Skill conflicts] in startup banner. Or run `python3 -c "import yaml; yaml.safe_load(open('.pi/skills/<name>/SKILL.md').read().split('---',2)[1])"` on any affected file.

## Root Cause Analysis

### Reproduce

Start Pi in the bigpowers repo → observe 23 [Skill conflicts] in banner. Or run `yaml.safe_load()` on any affected `.pi/skills/*/SKILL.md` frontmatter → `ScannerError` / `ParserError`.

### Isolate

All 30 affected files share the `description:` YAML frontmatter field. No other frontmatter fields cause errors. The 6 semantic-only files have `model: sonnet` as literal text inside description rather than as a separate YAML key.

### Hypothesize

Three confirmed hypotheses, verified by `yaml.safe_load()`:

1. **Inner double quotes** (17 files): `description: "text with "inner" quotes"` terminates the YAML string at the inner `"`.
2. **Broken `""` delimiter** (7 files): `description: ""...""` is not valid YAML string syntax.
3. **Embedded model hint** (6 files): `description: "...stuff.model: sonnet"` parses but Pi can't extract `model`.

### Verify

Corrected YAML (single-quote wrapper + separate `model:` key) passes `yaml.safe_load()` for all three patterns. See Phase 4 test above.

Two distinct YAML quoting antipatterns in the `description:` field of skill frontmatter:

**Pattern 1 — Broken `""` delimiter (7 files):**
Files use `description: ""..."" ` — two double-quote characters at both start and end, which is not valid YAML string syntax. In YAML, `"..."` is a quoted string; the extra `"` at each end creates an unparseable construct. Some of these files also embed `"model: sonnet"` inside the description instead of placing it as a separate YAML key.

**Pattern 2 — Unescaped inner double quotes (17 files):**
Files use `description: "text with "quoted speech" inside"` — the inner double quotes terminate the YAML string early, leaving trailing text as an unexpected scalar.

**Pattern 3 — Embedded model hint (6 files):**
Files parse successfully but `model: sonnet` appears as literal text inside the description string rather than as a separate YAML key. Pi cannot read the model hint from the description body.

The code paths involved: YAML frontmatter parsing during Pi skill loading. The failures occur because the YAML spec requires inner double quotes to be escaped (`\"`) or the outer wrapper to use single quotes. The `""..."" ` pattern is simply invalid YAML.

**Contributing factors**: No automated validation of SKILL.md YAML frontmatter before sync or commit. The `sync-skills.sh` script generates Cursor/Gemini artifacts but does not validate YAML structure.

**Risk level**: Medium — skills with broken frontmatter still load (the Markdown body renders), but Pi loses the `description` summary and `model` routing hint, degrading agent routing quality.

## Resolution

**Fixed:** 2026-06-18

**Root cause confirmed:** `scripts/sync-skills.sh` had three bugs: (1) greedy awk extraction leaked `model:` and later YAML fields into `$description`, (2) `echo "description: \"$description\""` didn't escape inner double quotes in generated YAML, (3) generated `.pi/skills/` files didn't preserve the `model:` key from source SKILL.md. Source files (`*/SKILL.md`) were fine — 61/62 parsed correctly.

**Fix applied:** 6 edits to `scripts/sync-skills.sh`:
- Bound description extraction to stop at next YAML key (prevents `model:` leakage)
- Added `sed 's/\\\\/\\\\\\\\/g; s/"/\\\\"/g'` escaping for double-quoted YAML output
- Extract and propagate `model:` key to generated `.pi/skills/` files

**Hardening added:** Regression guard in `sync-skills.sh` runs `scripts/validate-skill-yaml.py` after generation; exits 1 on any parse or semantic failure. Also added `scripts/validate-skill-yaml.py` as a standalone validation harness.

**Evidence:**
- `python3 scripts/validate-skill-yaml.py` → 62/62 parse OK, 0 semantic issues
- `bash scripts/sync-skills.sh` → 62 skills synced, guard passes
- `npm run compliance` → 88/88 PASS, 100% score

**Behavioral proof:** All 62 generated `.pi/skills/*/SKILL.md` files parse with `yaml.safe_load()`. All 62 preserve their `model:` key (38 `sonnet`, 14 `haiku`, 5 `opus`). No descriptions contain embedded model hints. Sync guard blocks regressions (tested: corrupt output → exit 1).

**Commit:** `fix(scripts): sync-skills.sh — escape YAML quotes, preserve model key, add validation guard`
