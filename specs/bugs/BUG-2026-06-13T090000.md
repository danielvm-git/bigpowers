# BUG-2026-06-13T090000: Doctrine drift — docs teach stale artifact names, dead skill references, and an undefined quality gate

## Problem

The doctrine layer (`docs/PRINCIPLES.md`, `docs/references/gates.md`, `docs/references/model-profiles.md`) has drifted from the current skills catalog and runtime spec layout. Five distinct inconsistencies were found during a consumer-side audit:

1. **Stale artifact names in gates.md + PRINCIPLES.md** — transition gates still require `PLAN.md`, `PROJECT.md`, `CONTEXT.md`, `SUMMARY.md`, `VERIFICATION.md`, `RESEARCH.md`, and `STATE.md`/`RELEASE-PLAN.md`. Every installed skill and the runtime actually use `state.yaml`, `release-plan.yaml`, and `specs/epics/` YAML shards. A downstream-generated `CONVENTIONS.md` inherits the wrong names and contradicts the YAML workflow agents actually follow.

2. **SKILL.md size rule violated by 12 skills (no enforcement)** — `docs/PRINCIPLES.md:100` mandates `SKILL.md < 100 lines`, but 12 of 61 skills currently exceed it (e.g., `seed-conventions` at 221 lines). No lint check enforces the rule; it exists only in prose.

3. **Dead skill names in model-profiles.md** — `docs/references/model-profiles.md` lists `refactor-code`, `debug-error`, and `orchestrate` in the model assignment tables. None of these directories exist. The correct names are `plan-refactor`, `diagnose-root`, and `orchestrate-project`.

4. **Hardcoded skill count is stale** — `model-profiles.md:83` says "expect 58" from `grep -rl '^model:' */SKILL.md | wc -l`. Actual count is **61**. The `using-bigpowers` skill and README badge also say 61 — one count is correct, two are wrong.

5. **94% quality threshold has no defined instrument** — `PRINCIPLES.md:67` and `gates.md:21-27` cite a 94% hard gate. The compliance harness (`scripts/audit-compliance.sh`) uses binary pass/fail per Gherkin scenario — it produces no numeric percentage. There is no scoring function that yields 94 vs 89.

**Expected behavior:** The doctrine layer accurately reflects the current YAML-first artifact layout, the live skill catalog, and a measurable quality gate.

## Root Cause Analysis

All five items share a single underlying cause: **facts hardcoded in prose diverge from the live catalog as skills evolve**.

- `docs/PRINCIPLES.md` and `docs/references/gates.md` were authored during or before the Markdown→YAML migration and never updated when `state.yaml` / `release-plan.yaml` / `specs/epics/` became canonical. No validation step checked that the docs matched the running workflow.
- `docs/references/model-profiles.md` was authored with pre-rename skill names (`refactor-code`, `debug-error`, `orchestrate`) and a count that was accurate at the time (58) but was never regenerated after three new skills were added. No CI check asserts that every name in the table resolves to a real skill directory.
- The 100-line SKILL.md rule exists only in prose; there is no automated lint that would have blocked the 12 over-length files from landing.
- The 94% threshold is an aspirational anchor copied from earlier documentation; no scoring function in the repo produces a numeric percentage from the Gherkin suite.

**Risk level: Medium** — does not break runtime behavior, but an agent reading the docs and following gate instructions will look for files that don't exist (`PLAN.md`, `STATE.md`, etc.) and skip YAML artifacts that do exist. Consumer-generated `CONVENTIONS.md` files ship this contradiction downstream.
