# BUG-2026-06-11T160000: `{dim}` and `{|}` render as literal text after TUI prototype adaptation

## Problem

**Actual behavior:** After the TUI prototype-adaptation commit (`fba0e49`), the dashboard renders `{dim}`, `{/dim}`, and `{|}` as visible literal characters throughout every panel — metrics bar, pipeline, STATE.YAML, epic queue, ledger. Example on screen:

```
{dim}BCPs:{/dim} 46{dim}/21{/dim}  {dim}|{/dim}
{dim}active_flow:{/dim} release
sur {dim}·{/dim} pla {dim}·{/dim} kic
```

**Expected behavior:** Labels and separators should render in muted gray; the separator dots in the stage tracker should be dim. No `{…}` characters should appear in the terminal output.

**Relation to prior bug:** This is a regression introduced by the prototype-adaptation work. BUG-2026-06-11T120000 (same scope) fixed the "all tags literal" case by adding `tags: true` to every box — that fix is still in place and unaffected. The new bug is a different root cause: newly-introduced `{dim}` tags are *themselves* invalid.

## Root Cause Analysis

Blessed's `_parseTags()` (in `element.js`) calls `program._attr(tagName)` for each `{tag}`. When `_attr()` returns `null` — meaning the tag name is unrecognised — the parser emits the raw `{tag}` string as literal output (line 499-501 of `element.js`).

The `_attr()` switch in `program.js` recognises exactly: `normal`, `default`, `bold`, `ul` / `underline` / `underlined`, `blink`, `inverse`, `invisible`, and the color family `{color-fg}` / `{color-bg}` (where color ∈ black, red, green, yellow, blue, magenta, cyan, white, grey, gray, light-grey, etc.).

**`dim` is not in this set.** It was never part of blessed's tag vocabulary. The prototype-adaptation code adopted `{dim}` as a shorthand for "muted label" throughout all render functions — but blessed silently passes it through as literal text.

A secondary issue: `{|}` used for right-alignment in the metrics bar is also not a blessed tag (`|` is not in the `[\w\-,;!#]` character class that the tag regex matches), so it renders as `{|}` literally.

**Correct replacement:**
- `{dim}text{/dim}` → `{gray-fg}text{/gray-fg}`
- `{|}` right-align marker → removed; right-aligning content must be done via manual string padding

**Scope:** All seven render files touched in the prototype-adaptation commit.

Risk level: Low — cosmetic/rendering only, no data layer changes.

## Resolution

**Fixed:** 2026-06-11

**Root cause confirmed:** `{dim}` is not a recognized blessed markup tag; when `_attr('dim')` returns `null`, `_parseTags()` emits raw `{tag}` text as literal output. Similarly, `{|}` contains `|` which is outside the tag-name regex character class.

**Fix applied:**
- Replaced `{dim}` → `{gray-fg}` and `{/dim}` → `{/gray-fg}` across all 7 TUI render files (metrics-bar.js, pipeline.js, state-yaml.js, epic-queue.js, ledger.js, filesystem.js, index.js)
- Removed `{|}` from the status line in metrics-bar.js

**Hardening added:** Test 5 in `smoke.test.js` now asserts all 6 TUI render functions do not emit literal `{dim}` or `{|}` substrings. This catches any future regression immediately.

**Evidence:** All 5/5 smoke tests pass (`cd dashboard && npm test`):
